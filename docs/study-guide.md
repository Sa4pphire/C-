# C++ 实习核心知识与 TaskHub 代码地图

## 现代 C++ 与 STL

- `std::string` 管理动态字符数据，避免手工分配与释放；见 `Task`。
- `std::vector` 表示连续、可变长序列；见任务列表和测试集合。
- `std::optional<Task>` 明确表达“任务可能不存在”；见查询和更新接口。
- 范围 `for` 用于遍历任务并统计；见 `TaskService::stats`。
- 结构化绑定用于拆开键值对；见测试运行器中的测试名称和函数。
- `<algorithm>` 配合迭代器表达通用操作；见空白标题检查。

复习时不要只背定义。对每个类型回答：谁拥有数据、是否允许为空、何时复制、复杂度是什么。

## RAII、指针与异常

RAII 把资源的生命周期绑定到对象生命周期。对象构造成功后资源有效；离开作用域时析构函数自动释放资源，即使中途抛出异常也一样。

- `unique_ptr`：唯一所有者，适合明确独占的资源。
- `shared_ptr`：多个对象共享所有权；需要注意循环引用和额外开销。
- 引用：对象必须存在且不表示所有权。
- 原始指针：通常表示非拥有访问，或对接 C API。

TaskHub 的 SQLite C API 返回原始句柄，仓储和 `Statement` 把它们包装成 RAII 生命周期。业务校验抛出 `ValidationError`，路由层统一映射成 `400`；数据访问异常映射成 `500`。

## CMake 与 Git

CMake 描述四类目标：核心业务库、HTTP 库、服务程序和测试程序。链接关系决定头文件与库如何传递。`TASKHUB_WARNINGS_AS_ERRORS` 用于质量检查，`TASKHUB_ENABLE_TSAN` 用于数据竞争检测。

一次合格的 Git 提交应当：主题单一、能够解释原因、差异可审查、测试状态清楚。常用顺序是：建立分支 → 修改与测试 → 查看 `diff` → 暂存相关文件 → 提交 → 查看日志。

## Linux 与排障

重点不是记住所有命令，而是能组合工具回答问题：文件在哪里、谁在运行、端口是否监听、日志有什么、上一步是否成功。优先掌握 `pwd`、`ls`、`find`、`grep`、管道、重定向、`ps`、`kill`、`chmod`、`ss` 和退出码。

## TCP、HTTP 与 JSON

服务器依次执行 socket、bind、listen、accept；每个连接通过 recv 读取字节，通过 send 写回字节。HTTP 在 TCP 字节流上定义请求行、请求头、空行和可选请求体。JSON 只是请求体的一种数据格式。

- `200`：成功查询或更新。
- `201`：成功创建。
- `204`：成功但没有响应体。
- `400`：客户端输入不合法。
- `404`：资源或路由不存在。
- `500`：服务器内部或数据库操作失败。

## SQL 与 SQLite

CRUD 分别对应 `INSERT`、`SELECT`、`UPDATE` 和 `DELETE`。事务让一组操作要么全部成功，要么全部失败。索引用额外存储和写入成本换取查询速度。TaskHub 使用参数绑定，避免引号处理错误和 SQL 注入。

## 线程与同步

竞态条件表示结果取决于不可预测的线程执行顺序。`mutex` 保护共享数据，`lock_guard` 在离开作用域时自动解锁。死锁常由多个锁以不同顺序获取产生。锁范围越大越安全但并行度越低；范围越小越高效但更难保证正确。

ThreadSanitizer 能发现运行时实际触发的数据竞争，但一次无报告不等于程序绝对没有并发缺陷。

## 测试层次

- 服务层测试：直接调用 `TaskService`，关注业务行为。
- HTTP 集成测试：通过 `ApiRouter` 输入请求并检查状态码与 JSON。
- 失败路径测试：验证非法输入、缺失资源和依赖错误。
- 并发测试：多个线程同时创建任务，检查没有丢失数据。

测试名称应描述行为。修复缺陷时先写能稳定失败的测试，再做最小修复，最后运行全部回归测试。
